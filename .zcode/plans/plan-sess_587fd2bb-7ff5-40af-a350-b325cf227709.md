## План: исправление поиска коллекций + экономия токенов

### Проблема 1: поиск по коллекции не работает

**Симптом:** пользователь пишет `spicedwine-42846`, бот пересылает в LLM, LLM не понимает что это slug подарка и не вызывает нужный tool.

**Корень:** нет парсинга slug-паттернов (`spicedwine-42846`, `scaredcat-3387`) до отправки в LLM. LLM должен сам догадаться, но это ненадёжно.

**Решение: два изменения**

1. **`bot/handlers/_chat_core.py` —新增 `detect_and_resolve_gift_slug()`**:
   - Regex для slug-паттернов: `[a-z]+-\d{1,7}` (без пробелов, с дефисом и числом)
   - Парсит `spicedwine-42846` → `model_name="spicedwine"`, `number=42846`
   - Вызывает `GiftWikiService` с новым методом `lookup_by_model_name("spicedwine")` (GiftWiki API поддерживает `model_name` параметр на `GET /collection`)
   - Если найдено — подменяет slug в user_text на каноническое имя коллекции: `Spiced Wine`
   - Если не найдено — оставляет как есть (LLM разберётся сам)

2. **`services/giftwiki_service.py` —新增 `lookup_by_model_name()`**:
   - `GET /collection?model_name=spicedwine&limit=5`
   - Возвращает `[{id, name, count, startapp}]`
   - Кешируется как `oclp:giftwiki:model:<model_name>` (TTL 1ч)
   - Использует существующую проекцию `_project_collection_item()`

**Файлы изменений:**
- `services/giftwiki_service.py` — новый метод `lookup_by_model_name()`
- `bot/handlers/_chat_core.py` — новый `detect_and_resolve_gift_slug()`, вызывается в `generate_reply()` перед LLM

### Проблема 2: высокий расход токенов

**Симптом:** ~52 КБ системного промпта (instruction.md 41 КБ + 4 скилла 11 КБ) + tool schemas отправляются с каждым запросом. Для простых вопросов «цена пепе» это ~10-12K входных токенов только на системный контекст.

**Решение: ленивая подгрузка скиллов (lazy skills)**

1. **`services/llm_service.py` — `_build_dynamic_system_prompt()`**:
   - Всегда отправлять только `instruction.md` (41 КБ) — он содержит описание того, какие скиллы существуют и когда их использовать
   - Скиллы (`slang.md`, `gift-numbers.md`, `market-prices.md`, `orig-accounts.md`) хранить отдельно в памяти
   - Метод `_should_include_skill(user_text)` — определяет по ключевым словам/паттернам, какие скиллы нужны:
     - `market-prices` — по словам: флор, цена, floor, сколько стоит, грам, cheaper, выгоднее, маркеты, grapes, mrkt...
     - `slang` — по словам: сленг, кто такой, что такое, autoGifts, p2p, скам, рейд...
     - `gift-numbers` — по словам: тир, номер, tier, ценный номер, зеркальный...
     - `orig-accounts` — по словам: ориг, аккаунт, релеер, original...
   - Если ни один скилл не подошёл — отправлять только instruction.md без скиллов

2. **`services/llm_service.py` — изменения в конструкторе и `chat()`**:
   - Конструктор принимает `skills: dict[str, str]` (имя → контент)
   - В `chat()`: собирает динамический system_prompt на основе user_text

3. **`main.py` — `_init_llm()`**:
   - Загружает instruction.md отдельно, скиллы отдельно
   - Передаёт скиллы в LLMService конструктор

**Файлы изменений:**
- `services/llm_service.py` — новый `_build_dynamic_system_prompt()`, `_should_include_skill()`, изменение конструктора
- `main.py` — разделение загрузки instruction.md и skills

---

### Порядок реализации

1. `services/giftwiki_service.py` — метод `lookup_by_model_name()`
2. `bot/handlers/_chat_core.py` — `detect_and_resolve_gift_slug()` + интеграция в `generate_reply()`
3. `services/llm_service.py` — ленивая подгрузка скиллов
4. `main.py` — разделение загрузки промпта и скиллов
5. Тестирование: проверка что slug-паттерн корректно парсится, GiftWiki API отвечает, скиллы правильно подгружаются

### Что НЕ меняем
- instruction.md не трогаем (оставляем на будущее, если понадобится сжать)
- Tool schemas остаются всегда (они компактные ~3 КБ)
- История диалогов остаётся как есть
