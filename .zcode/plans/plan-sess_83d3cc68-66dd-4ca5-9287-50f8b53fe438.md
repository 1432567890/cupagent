## Что нашёл (живые запросы с нашим токеном)

**GetGems клиент сломан в двух местах**, GiftWiki — уже работает корректно.

1. `fetch_floor_prices()` стучится на `/v1/gifts/collections`, который реально НЕ отдаёт `floorPrice` и не имеет вложенного объекта `collection` → код молча собирает 0 цен → LLM говорит «нет данных с гетгемс».
2. `fetch_collection_attributes()` — заглушка, всегда возвращает `{models:[], backdrops:[]}`. Но у GetGems есть полноценный `/v1/collection/attributes/{address}` с флорами по Model/Backdrop/Symbol.

**GiftWiki** (api.giftwiki.tg): проверил `/collection`, `/gifts/monochromes`, `/collection?search=` — все отвечают 200, структура полностью совпадает с тем, что уже ожидает `giftwiki_service.py`. **Менять нечего.**

## План правок (только `markets/getgems/client.py`)

### 1. `fetch_floor_prices()` — перейти на `/v1/gifts/collections/top?kind=all`
- Топ-эндпоинт отдаёт `{place, collection:{address,name,…}, floorPrice}` + курсор вида `"18:10"`, по 10 за страницу.
- Проходим страницы пагинации (курсор через query-параметр `after`), пока курсор не пустой, с тем же хард-капом `_MAX_PAGES` (20 → до 200 коллекций, достаточно).
- Извлекаем `name` и `floorPrice` (float, уже в TON — GetGems = TON 1:1, как и было).
- **Побочный эффект**: попутно строим `self._name_to_address: dict[str, str]` (имя → адрес коллекции). Он нужен шагу 2. Это поле экземпляра, обновляется при каждом цикле обновления цен.

### 2. `fetch_collection_attributes(collection_name)` — реальная реализация
- Резолвим имя в адрес через `self._name_to_address`. Если маппинга нет (цен ещё не подтягивали / имя не из топа) — один раз вызываем `/v1/collection/basic-info/{address}`… нет, без адреса не вызвать. Поэтому: если адреса нет, сначала прогоняем полный цикл top-пагинации только ради мэппинга (дешево — это те же запросы, что и для цен). Чтобы не плодить логику — вынесу построение мэппинга в хелпер `_build_name_to_address()`, который вызывается из обоих мест и заполняет `self._name_to_address`, если он пуст.
- С адресом → `GET /v1/collection/attributes/{address}` → ответ `{response:{attributes:[{traitType, values:[{value, count, minPrice, minPriceNano}]}]}}`.
- Проекция в существующий контракт `{"models": [...], "backdrops": [...]}` (каждый элемент `{name, price}`), как у MRKT/Portal — `traitType == "Model"` → models, `"Backdrop"` → backdrops. `Symbol` не возвращаем (контракт других клиентов — только models+backdrops; не расширяю без явной нужды).
- `minPrice` парсим через `float()` (это строка, напр. `"22.8"`). `minPriceNano` игнорируем — `minPrice` уже удобная TON-форма.

### 3. Что НЕ трогаю
- `base_url` (`https://api.getgems.io/public-api`) — корректный, проверено: `/public-api/v1/gifts/collections` отвечает 401 без токена и 200 с ним.
- `_auth_headers`, `authenticate()` — Bearer работает как есть.
- `giftwiki_service.py` — уже корректно работает с реальным API.
- `price_service.py`, `llm_service.py`, `main.py` — `GetGemsClient` уже зарегистрируется, когда задан `GETGEMS_API_TOKEN` (он задан). После фикса клиента цены реально потекут, и LLM перестанет говорить «нет данных с гетгемс».

## Проверка после правки
- Запустить `python -c "asyncio.run(...)"` мини-скрипт, который инстанцирует `GetGemsClient(token)`, вызывает `authenticate()` → `fetch_floor_prices()`, и печатает количество ненулевых цен + пару примеров (Scared Cat, Plush Pepes). Ожидаю: десятки коллекций с реальными `floorPrice`.
- Вызвать `fetch_collection_attributes("Scared Cat")` → ожидаю непустые `models` и `backdrops` с ценами.
