# План: Crypto/Fiat + GiftWiki + Telegram custom emoji

## Цель (из запроса)
1. Добавить инструменты для конвертации крипты в фиат (`Binance` + `CBR`/`ExchangeRate`) — чтобы бот отвечал «что можно купить на X рублей из подарков?», «5 GRAM в рублях», «1 биткоин в рублях».
2. Добавить интеграцию `GiftWiki API` (`X-API-Key` из `.env`) — поиск коллекций, деталей, monochrome-классификации.
3. Распарсить локально emojis из `api.changes.tg` и использовать нативные Telegram `<tg-emoji emojiId="...">` когда бот упоминает коллекцию/модель/фон.
4. **Минимизировать токены** во всём — компактные схемы, ленивая загрузка, кэш в Redis.

## Выбор пользователя (подтверждено)
- **Crypto API:** Binance `/ticker/price` (крипто→USDT) + open.er-api.com (USD→фиат, fallback на ЦБ `cbr-xml-daily.ru`).
- **Telegram emoji:** нативный `customEmojiId` через changes.tg (без стикерпаков).

---

## Архитектура (по AGENTS.md)

Новые сервисы добавляются как **отдельные модули** в `services/`, каждый за собой: Protocol/абстракция → сервис с DI через kwargs → регистрация в `dp["..."]` в `main.py` → использование из handler'а и LLM-инструментов.

LLM tool-calling уже расширен (`_MAX_TOOL_ROUNDS=3`), нужно:
- добавить **новые инструменты** в `payload["tools"]` (массив),
- сделать **универсальный dispatcher** в `_execute_tool_call` (сейчас захардкожен только `get_floor_prices`).

## Файлы для создания

### 1. `services/crypto_service.py` (НОВЫЙ)
Кэшированные курсы крипты и фиата. TTL в Redis 60с (крипто) / 6ч (фиат).

- **Protocol** `CryptoPriceProvider` (абстракция за Binance) — на будущее для тестов/фолбэка.
- **`CryptoService`** с DI: `redis`, `binance_base_url`, `fiat_base_url`, `cbr_url`, `binance_session`, `fiat_session`.
- Методы (всё async):
  - `get_crypto_prices(symbols: list[str]) -> dict[str, Decimal]` — Binance batch `/ticker/price?symbols=[...]`. Символы: `BTCUSDT`, `ETHUSDT`, `TONUSDT`, `GRAMUSDT` (если есть). Сначала Redis → мимо → запрос → кэшируем 60с.
  - `get_fiat_rates(base="USD") -> dict[str, Decimal]` — open.er-api.com → Redis 6ч → fallback на `cbr-xml-daily.ru/daily_json.js`.
  - `convert(amount: Decimal, from_coin: str, to_currency: str) -> Decimal` — crypto×fiat.
  - `close()`.
- Словарь алиасов: `{"btc":"bitcoin","bitcoin":"BTCUSDT","тон":"TONUSDT","ton":"TONUSDT","gram":"GRAMUSDT","грам":"GRAMUSDT","eth":"ETHUSDT", ...}`.
- В Redis ключи: `oclp:crypto:prices` (hash), `oclp:fiat:USD`.

### 2. `services/giftwiki_service.py` (НОВЫЙ)
Кэшированный клиент GiftWiki. TTL: список коллекций 10мин, детали коллекции 1ч, monochrome 1ч.

- **`GiftWikiService`** с DI: `api_key`, `base_url`, `redis`, `aiohttp_session`.
- Заголовок: `X-API-Key: <key>`.
- Методы:
  - `search_collections(query: str, limit=10) -> list[dict]` → `GET /collection?search=...&limit=...` (фильтруем поля: `_id, name, count, startapp`).
  - `get_collection_detail(id: str) -> dict` → `GET /collection/{id}` (фильтруем: name, keywords, gifts[name+model_name+rarity+supply_tier], related_collections[name]).
  - `get_monochromes(gift_name=None, model_name=None, type=None) -> list[dict]` → `GET /gifts/monochromes?...`.
- Все ответы **проецируются в компактный формат** (выкидываем `photo_url`, `animation_url`, `preview_gifts`, `created_at`), чтобы экономить токены в LLM-ответе.
- Redis-кэш с TTL, ключ `oclp:giftwiki:search:{query}`.
- `close()`.
- Если `api_key` пустой → сервис не создаётся (`None`), инструмент не регистрируется.

### 3. `services/emoji_registry.py` (НОВЫЙ)
Локальная in-memory карта `giftId/customName → customEmojiId`, загружаемая один раз при старте.

- **`EmojiRegistry`** с DI: `base_url="https://api.changes.tg"`, `local_cache_path`, `redis`, `aiohttp_session`.
- Данные (спарсить локально):
  - `GET /ids` → `{giftId: giftName}` (≈116 записей).
  - `GET /emoji` → `{giftId: customEmojiId}` (≈158).
  - Локально сохраняем в `data/changes_tg_emojis.json` (создаётся скриптом #6).
- На старте: загрузить из локального файла → если файла нет, fetch с changes.tg и сохранить.
- Опционально периодическое обновление (раз в сутки, фоновая таска).
- Методы:
  - `get_emoji_id_by_gift_name(name: str) -> str | None` — поиск по giftName (case-insensitive, с транслитерацией ру→лат).
  - `get_emoji_id_by_gift_id(gift_id: str) -> str | None`.
  - `format_emoji_html(emoji_id: str, fallback_text: str) -> str` → `<tg-emoji emoji-id="ID">emoji</tg-emoji>`.
- `close()`.
- **Важно:** TelegramBot API поддерживает `<tg-emoji emoji-id="...">` только в `parse_mode=HTML` (что уже стоит `DEFAULT_PARSE_MODE="HTML"`).

### 8. `services/llm_service.py` (ИЗМЕНИТЬ — критично)
**Унификация dispatcher'а инструментов.** Сейчас захардкожен `get_floor_prices` — нужно обобщить.

Рефактор:
- Список инструментов собирается динамически из доступных сервисов:
  ```python
  def _build_tools(services) -> list[dict]:
      tools = []
      if services.get("price_service"): tools.append(_FLOOR_PRICES_TOOL)
      if services.get("crypto_service"): tools.append(_CONVERT_TOOL)
      if services.get("giftwiki_service"): tools.append(_GIFTWIKI_SEARCH_TOOL, _GIFTWIKI_DETAIL_TOOL)
      return tools
  ```
- Новые константы-схемы (компактные, минимум описаний):
  - `_CONVERT_TOOL` (`convert_currency`):
    ```
    params: { amount: number, from: string (btc/ton/gram/usd/rub...), to: string }
    description: "Конвертировать крипту/фиат. Например: 5 GRAM в рублях, 1 BTC в RUB."
    ```
  - `_GIFTWIKI_SEARCH_TOOL` (`search_collections`):
    ```
    params: { query: string, limit: int=10 }
    ```
  - `_GIFTWIKI_DETAIL_TOOL` (`get_collection_detail`):
    ```
    params: { id: string }
    ```
- Метод `chat()` принимает `services: dict` вместо одного `price_service`.
- `_execute_tool_call(tc, services)` — switch по имени функции:
  ```python
  if fn == "get_floor_prices": ...
  elif fn == "convert_currency": ...
  elif fn == "search_collections": ...
  elif fn == "get_collection_detail": ...
  ```
- Все tool-results в компактном JSON (выкидываем лишнее).
- **Обратная совместимость:** оставить параметр `price_service` как алиас, который маппится в `services={"price_service": price_service}`.

## Минимизация токенов (ключевые меры)

1. **Схемы инструментов компактные** — описание 1-2 предложения, без избыточных полей.
2. **Tool-results проецируются** — выкидываем `photo_url`, `animation_url`, `preview_gifts`, `created_at`, `telegram_url` (это сотни байт на коллекцию).
3. **Redis-кэш везде** — крипто 60с, фиат 6ч, giftwiki 10мин-1ч, emoji — локальный файл.
4. **`temperature=0`** уже стоит — ответы детерминированные.
5. **System-prompt дополнения** — короткие секции, без длинных таблиц.
6. **Emoji registry in-memory** — загружается один раз, не передаётся в LLM (используется только при рендере ответа, не влияет на токены).

## Риски / заметки

- **`GRAM` на Binance:** может не быть пары `GRAMUSDT` (новая монета). Если 404 — fallback к `TONUSDT` (равны 1:1) с предупреждением в логе.
- **`<tg-emoji>` рендеринг:** работает только если бот добавлен в стикер-набор **или** emojiId из публичного набора. changes.tg публикует emoji из публичных стикер-паков GiftChanges — рендер у юзеров должен работать. Если нет — текст показывается как fallback (не ломает UX).
- **Binance rate limit:** 1200 весов/мин — для нас копейки. ЦБ обновляется раз в сутки — кэшируем 6ч, норм.
- **changes.tg attribution:** API требует видимый кредит @GiftChangesAPI. Добавлю в `_GHOST_FOOTER` или отдельным «данные emoji: @GiftChangesAPI». Решу при реализации.
- **Без тестов в этом плане** — пользователь не просил, но структура (Protocol/DI) позволяет легко добавить pytest-asyncio позже.
- **Обратная совместимость:** параметр `price_service` в `chat()` остаётся как алиас, чтобы не сломать существующих вызовов.
