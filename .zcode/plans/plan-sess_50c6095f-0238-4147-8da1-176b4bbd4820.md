## Часть 2: Стиль segment (на основе примеров)

Проанализировал примеры пользователя. Ключевые черты:
- **Короткие, обрывистые сообщения** («не знаю.», «ну хз смогу ли сделать»)
- **Слова-паразиты:** «ну», «короче», «че», «типа», «хз», «как-то», «там», «вроде», «короче говоря»
- **Сокращения/сленг:** мркт (вместо MRKT), порталс, грейпс, флор (вместо floor), лор, плюшевый (plush)
- **Регистр:** почти всегда строчные, минимум заглавных (даже в начале предложения)
- **Пунктуация:** часто без точки, иногда точка есть («не знаю.»), многоточия
- **Смайлики:** `^_^`, `о.О`, без избытка
- **Тон:** прямолинейный, прагматичный, слегка неуверенный («но надеюсь да чето сделаю», «ну хз»)
- **Техническая речь:** без ExplanationFluff, прямо по делу

### 2.1 `skills/instruction.md` — полная переработка секции «Стиль общения»
Заменю абстрактное «пиши естественно и непринуждённо» на конкретные правила + few-shot примеры из переписки segment:

```markdown
## Стиль общения (как segment)

Пиши как segment — владелец бота. Примеры его стиля:

> ну потому что это логично, эмодзи подарков/моделей в отдельных паках
> есть такая важная информация
> короче. в клиенте показывается реакция 🤣, но на самом деле есть реакция 😂 еще
> он знает актуальную цену флор подарков с мркт порталс и грапсов
> +лор подарков
> ну хз смогу ли сделать
> но надеюсь да чето сделаю

Правила:
- Строчные буквы почти везде, заглавная только если без неё нечитаемо
- Коротко, по делу. Объяснение только если реально нужно
- Слова: ну, короче, че, типа, хз, как-то, там, вроде, норм
- Сокращения: мркт, порталс, грейпс, флор, лор, плюшевый (plush)
- Без воды, без «как консультант», без «как я могу помочь»
- Если не уверен — пиши «хз», «вроде», «не знаю»
- Точка в конце — редко, многоточия норм
- Смайлики типа ^_^ уместны, но не на каждое сообщение
```

## Часть 3: Guest thread-контекст (reply chain, макс 30, TTL 3 дня)

**Подход:** по guest_query_id цепочки через `reply_to_message`. Telegram flatten-ит `reply_to_message` до 1 уровня, поэтому храним в Redis маппинг `message_id → thread_id`.

### 3.1 Новый модуль `bot/handlers/_guest_thread.py`

```python
_GUEST_THREAD_KEY = "oclp:guest_thread:{thread_id}"
_GUEST_MSG_INDEX_KEY = "oclp:guest_msg:{chat_id}:{message_id}"  # msg_id → thread_id
_THREAD_TTL = 3 * 24 * 3600  # 3 дня
_MAX_THREAD_MESSAGES = 30

async def resolve_or_create_thread(redis, message) -> str:
    """Определить thread_id для guest сообщения.
    
    Логика:
    1. Если message.reply_to_message есть — взять его message_id,
       найти thread_id через _GUEST_MSG_INDEX_KEY
    2. Если нашли — это продолжение треда, вернуть thread_id
    3. Если не нашли (новый тред) — создать новый thread_id (uuid4)
    4. Если reply_to_message нет — новый thread_id
    """

async def load_thread_history(redis, thread_id) -> list[dict]:
    """Загрузить контекст треда (макс 30 сообщений) из Redis."""

async def append_to_thread(redis, thread_id, chat_id, msg_id, role, content, tool_results=None):
    """Добавить сообщение в тред.
    - Сохраняет msg_id → thread_id маппинг
    - Добавляет {role, content} в список (с tool_results если есть)
    - Trim до последних 30
    - Обновляет TTL
    """
```

### 3.2 `bot/handlers/guest_chat.py` — использовать thread-контекст

```python
@router.guest_message(F.text.func(is_free_text))
async def handle_guest_chat(message, bot, llm, redis, price_service=None):
    # 1. Определить thread
    thread_id = await resolve_or_create_thread(redis, message)
    
    # 2. Загрузить контекст треда
    history = await load_thread_history(redis, thread_id)
    
    # 3. Сгенерировать ответ с этим контекстом (а не с user_id-based)
    reply = await llm.chat(
        user_text, user_id=user_id, history=history,
        price_service=price_service, on_thread_save=...,
    )
    
    # 4. Сохранить user message + assistant reply в тред
    await append_to_thread(redis, thread_id, chat_id, message.message_id, "user", user_text)
    # assistant reply получит свой msg_id после answer_guest_query (SentGuestMessage.inline_message_id)
    # но inline_message_id ≠ обычный message_id — для reply chain используем guest_query_id
```

**Проблема с msg_id бота:** `answer_guest_query` возвращает `SentGuestMessage` с `inline_message_id` (строка), а не числовой `message_id`. Значит когда пользователь reply-ит на ответ бота, `reply_to_message.message_id` в приходящем guest_message будет числовым ID этого сообщения в чате. Нужно сохранять этот ID.

**Решение:** после `answer_guest_query` мы НЕ знаем message_id, который получит сообщение (оно только отправляется). Но когда придёт следующий guest_message с reply_to_message, там будет `reply_to_message.message_id` — это ID сообщения бота. Чтобы связать: когда сохраняем assistant reply, используем `guest_query_id` как «предполагаемый» message_id. При следующем guest_message смотрим `reply_to_message.message_id` → если его нет в index → создаём новый тред.

**Уточнённый подход:** индекс `msg_id → thread_id` строится лениво:
- Для user messages: `message.message_id → thread_id` (точно знаем)
- Для bot replies: мы не знаем message_id заранее, но когда пользователь reply-ит, мы видим `reply_to_message.message_id` в новом сообщении. Если этот ID уже в индексе (от предыдущего user msg) — продолжаем тред. Если нет — создаём новый.

Это работает потому что: в одном треде пользователь всегда видит цепочку сообщений, и reply_to_message будет указывать либо на ответ бота (новый msg_id, которого нет в индексе) либо на другое сообщение пользователя (есть в индексе).

**Простой fallback:** Если `reply_to_message` есть, но его msg_id не в индексе — всё равно создаём/продолжаем тред, привязанный к chat_id. Т.е. thread_id = `chat_id:{chat_id}:{first_msg_id_in_chain}`. На практике в бизнес-чате пользователь обычно общается в одном потоке.

### 3.3 Контекст включает tool results
`append_to_thread` хранит не только `{role, content}`, но и результаты tool-calls (если были). В LLMService добавлю возврат информации о том, какие tools вызывались — или проще: после получения reply, если reply содержит цены, добавим в контекст отдельное сообщение `{"role": "system", "content": "[бот показал цены с маркетов]"}`.

Проще: хранить в контексте пары user+assistant как обычно, а результаты tool_calls включаются автоматически в `messages` внутри `LLMService.chat()` (они уже добавляются как `role: "tool"`). Но эти tool messages не сохраняются в history. 

**Решение:** в `generate_reply_for_guest` после вызова LLM сохраним в thread только `{role: "user", content: user_text}` и `{role: "assistant", content: reply}`. Tool results теряются между сессиями — это ок, ИИ при новом запросе вызовет tool заново если нужно.

### 3.4 `LLMService` — не менять, использовать как есть
`llm.chat(history=...)` уже принимает history и добавляет к нему. Просто guest handler будет передавать thread-history вместо user-history.

## Риски / компромиссы

- **Стиль:** ИИ может не идеально копировать стиль — depends от модели. Few-shot примеры должны помочь.
- **Коллекции без словаря:** ИИ может ошибаться при сопоставлении сленга. Это компромисс — пользователь выбрал «сам угадывает».
- **Guest thread msg_id:** нет 100% надёжного способа связать bot reply с message_id. Используем эвристику: reply_to_message.message_id + chat_id. В худшем случае — каждый новый визит = новый тред (теряется контекст), что приемлемо.
- **3 дня TTL:** длинные треды могут обрезаться. Макс 30 сообщений = ~15 пар user/assistant.
